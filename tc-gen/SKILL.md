# tc-gen：新需求测试用例生成（MCP增强版）

为 shcj 项目的新需求生成高质量测试概要与测试用例。**分 5 个阶段，每个阶段产出后必须停下来等用户确认/纠正，禁止一次性走完。**

本技能是 `testcase-gen` 的 MCP 增强版：原 skill 以本地文档为主；本版在本地文档基础上，强制使用 `user-dev-standards` MCP 做知识发现、需求校验和交叉验证。若 MCP 不可用或没有匹配结果，必须在阶段产物中说明原因。

## 核心原则

- **需求文档只写「增量」，`.dev-standards` 写「全量」。** 必须结合两者：既测新功能，又不漏被影响的老功能。
- **MCP 用于发现和校验，不替代全文阅读。** MCP 先帮助定位业务规则、需求疑点和服务知识；最终仍要读取本地业务文档、代码地图、流程定义和接口文档全文。
- **接地气且可追溯。** 所有测试点必须有证据链：需求变更项、MCP业务规则、业务文档章节、状态机/配置矩阵/通知矩阵、接口字段、代码地图 Handler/风险点。禁止编造泛泛用例。
- **人工卡点。** 每阶段结束输出「请确认是否继续」，等用户回复后再进入下一阶段。
- **版本隔离。** 每个迭代版本独立放在 `.test-standards/<版本>/` 下；`.test-standards` 根目录只初始化一次，不在根目录直接放 `input/output`。
- **严禁以活源码替代知识库。** `codemap`/业务文档/`procdefs` 均指 `.dev-standards` 下预先沉淀的**静态文档**，不是项目活源码。测试用例的依据是「系统应该做什么」（需求文档 + 接口规范），不是「系统现在写了什么」（活源码）。原因：① 同步开发的新功能代码在开发人员本机，远程仓库尚未有最新提交，读到的是不完整的早期骨架；② 用代码反推测试用例本质上是「以代码校验代码」，无法发现实现偏差。

## 调用方式与初始化规则

当用户说类似 `tc-gen初始化版本“V1.12.0-xxx”`、`tc-gen 初始化版本 V1.12.0-xxx` 时，执行初始化，不进入阶段0。

初始化规则：

1. 在代码项目根目录下检查 `.test-standards`。
2. 如果 `.test-standards` 不存在，则创建；如果已存在，不重复初始化根目录，不覆盖其中已有文件。
3. 初始化版本目录时，必须使用本 skill 内置 UTF-8 脚本，禁止用 PowerShell/cmd 直接 `mkdir` 或 `New-Item` 创建含中文版本目录：

```bash
python ".cursor/skills/tc-gen/scripts/init_version_utf8.py" "<项目根目录>" "<版本>"
```

4. 如果 `<版本>` 包含中文、空格或 shell 中显示乱码，先把版本名写入 UTF-8 文本文件首行，再使用 `--version-file`，避免 Windows GBK/CP936 终端把中文参数解码错误：

```bash
python ".cursor/skills/tc-gen/scripts/init_version_utf8.py" "<项目根目录>" --version-file "<UTF-8版本名文件>"
```

5. 脚本会在 `.test-standards` 下创建版本目录：`.test-standards/<版本>/`。
6. 在版本目录下创建：

```text
.test-standards/<版本>/
├── input/
│   ├── prodword/      # 需求文档目录，沿用原 requirement 逻辑
│   │   └── prodword_pic/  # 需求文档图片目录，Word 转 md 时同步抽取图片
│   └── reference/     # 三方接口文档/联动参考资料，沿用原 reference 逻辑
└── output/            # 阶段产物目录，沿用原 output 逻辑
```

7. 如果版本目录已存在，只补齐缺失子目录，不删除、不覆盖已有输入或输出。
8. 初始化完成后，提示用户把需求文档放到 `input/prodword/`，把接口/三方/联动资料放到 `input/reference/`。`input/prodword/prodword_pic/` 由转换流程存放需求文档图片，用户也可手动补充流程图原图。
9. `.test-standards` 根目录下不得直接创建版本无关的 `input/` 或 `output/`。

### 初始化编码要求

- 所有版本目录初始化必须走 `tc-gen/scripts/init_version_utf8.py`。
- 含中文版本名时优先使用 `--version-file`，版本名文件必须以 UTF-8 或 UTF-8 with BOM 保存。
- 初始化脚本只创建目录，不覆盖已有输入文件和阶段产物。
- Agent 不得用 shell 内嵌中文命令创建版本目录，例如 `mkdir ".test-standards/V1.0.0-支付宝"`、`New-Item`、`python -c "..."`。

### 版本输入发现要求

- 阶段0开始前，必须先运行本 skill 内置 UTF-8 脚本发现版本目录和输入文件，禁止用 `Glob` 直接扫描 `.test-standards/<中文版本>/`：

```bash
python ".cursor/skills/tc-gen/scripts/list_version_inputs_utf8.py" "<项目根目录>" "<版本>"
```

- 如果 `<版本>` 含中文、空格或 shell 中显示乱码，优先使用 `--version-file`；如果只有前缀是 ASCII 且能唯一匹配，也可以用 `--version-prefix`：

```bash
python ".cursor/skills/tc-gen/scripts/list_version_inputs_utf8.py" "<项目根目录>" --version-file "<UTF-8版本名文件>"
python ".cursor/skills/tc-gen/scripts/list_version_inputs_utf8.py" "<项目根目录>" --version-prefix "V1.0.0-saas"
```

- 脚本输出 JSON，包含 `versionRoot`、`input/prodword`、`prodword_pic`、`input/reference`、`output` 的存在性和文件清单。后续读取、转换、图片解析必须以该 JSON 中的绝对路径为准。
- 若 `Glob` 返回 0 个文件但脚本能找到文件，以脚本结果为准；不得据此判断用户未初始化或未放文件。

## MCP 使用规则

当前项目优先使用 `user-dev-standards` MCP。调用任何 MCP 工具前，必须先读取对应工具 descriptor/schema 文件，确认参数后再调用。

常用 MCP 工具：

- `get-business-rule`：阶段1查询候选业务规则文档。建议先列表，再按关键词匹配。参数建议带 `projectPath=d:\Applications\IDEA\shcj`。
- `validate-requirement`：阶段0/阶段2前校验需求文档质量，关注可测试性、流程闭环、状态流转、边界异常、接口与交互、影响评估。
- `get-service-knowledge`：阶段2/阶段3查询服务知识，补充服务职责、接口边界、幂等、异常处理和风险点。
- `query-standard`：仅在涉及代码评审、异常处理、事务、日志、数据库等规范风险时使用。

如果没有调用 MCP，必须在产物中写明：

```markdown
## MCP使用情况

- 未调用/未使用原因：例如 MCP 不可用、无匹配业务规则、当前阶段只基于用户确认后的旧产物修订。
- 风险提示：未经过 MCP 交叉校验，可能遗漏最新业务规则或服务知识。
```

## 输入输出约定

- 版本根目录：`.test-standards/<版本>/`，例如 `.test-standards/V1.12.0-xxx/`。
- 输入：`.test-standards/<版本>/input/prodword/`（需求文档，沿用原 requirement 逻辑）、`.test-standards/<版本>/input/prodword/prodword_pic/`（需求文档图片，尤其流程图）、`.test-standards/<版本>/input/reference/`（三方接口文档等联动文档）。
- 输出：`.test-standards/<版本>/output/` 下，按阶段写文件：`01-定位.md`、`02-变更分析.md`、`03-测试概要.md`、`04-测试用例.md`。
- 本地知识库（只读）：`shcj-common/.dev-standards/knowledge/business/`、`knowledge/codemap/`、`knowledge/service/`、`shcj-common/.dev-standards/procdefs/`。
- MCP知识源：`user-dev-standards` 下的业务规则、服务知识、需求校验、代码规范工具。
- 用例格式使用本技能内置的阶段4模板，固定字段为：`用例等级`、`所属模块`、`用例标题`、`前置条件`、`用例步骤`、`预期结果`。其中 `前置条件` 非必填；执行 `tc-gen` 时不需要额外读取 `.test-standards/templates/测试用例模板.md`。
- 详细的 Word 转换、判定表、状态迁移、接口测试、功能操作偏离、用例裂变规则沿用 `../testcase-gen/reference.md`。

## 阶段进度清单

```markdown
- [ ] 阶段0 准备：Word 转 md + 读取输入 + MCP需求校验
- [ ] 阶段1 定位：MCP匹配业务规则 + 读取业务文档/codemap/procdefs
- [ ] 阶段2 变更分析：需求 vs 业务现状 vs MCP服务知识
- [ ] 阶段3 测试概要：通过前置门禁后按6维度列测试点
```

## 阶段0：准备（读取输入 + MCP需求校验）

1. 明确当前处理的 `<版本>`；若用户未给出版本，先询问，不要把产物写到 `.test-standards` 根目录。
2. 使用 `tc-gen/scripts/list_version_inputs_utf8.py` 列出 `.test-standards/<版本>/input/prodword/`、`.test-standards/<版本>/input/prodword/prodword_pic/` 和 `.test-standards/<版本>/input/reference/` 下的所有文件；Windows/中文路径场景禁止用 `Glob` 的空结果作为缺文件结论。
3. 对每个 `.docx` / `.doc` 转换为同名 `.md` 放回 `input/prodword/`（保留原文件），并同步抽取 Word 内图片到 `input/prodword/prodword_pic/`。转换后的 md 中图片引用应尽量指向 `prodword_pic/` 下的文件；无法自动建立引用时，必须在 md 附近标注 `[图片：prodword_pic/<文件名>，需人工/AI查看]`。
4. 读取需求文档 + 全部联动文档 + `prodword_pic/` 下图片。三方接口文档特别留意：接口入参/出参/异常码/超时/重试/幂等。
5. 解析需求 md 时，搜索“流程图”“业务流程”“流程”“时序图”“泳道图”“状态流转”“节点”“步骤”等字眼，定位其附近图片引用；对这些图片必须重点查看并提炼流程节点、角色、分支、状态迁移、接口调用顺序和异常路径。图片内容与正文冲突时，单独列为疑点。
6. 调用 MCP `validate-requirement` 校验需求质量。若工具无入参读取能力，则先读取需求文档，由 agent 根据返回 prompt/规则完成审计。
7. 输出「读到的文档、图片/流程图清单、需求核心、MCP需求校验结论、待确认疑点」，停下来确认。

阶段0产物/回复必须包含：

- 输入文件清单。
- 图片清单：`prodword_pic/` 下图片总数、已重点解析的流程图/疑似流程图、未能解析的图片。
- 需求核心摘要。
- 流程图解析摘要：从图片提取出的主流程、角色、节点、分支和状态流转。
- MCP需求校验：错误/警告/建议。
- 测试范围待确认项，例如是否覆盖个人CA、存量数据、签章异常、静默签章、接口字段级。

## 阶段1：定位（MCP匹配业务流程）

1. 先调用 MCP `get-business-rule`：
   - 第一次用 `type=list` 或空值列出可用业务规则文档。
   - 第二次用需求关键词匹配候选流程，例如 `CFCA CA 建档 建档变更`。
   - 必须传入 `projectPath`。
2. 根据 MCP 候选和需求内容，匹配 `shcj-common/.dev-standards/knowledge/business/` 下对应业务流程文档。
3. 读取匹配到的本地业务文档全文 + 对应 `codemap` 代码地图 + 必要的 `procdefs` 流程定义。
4. 输出到 `.test-standards/<版本>/output/01-定位.md`：
   - MCP查询关键词、命中文档、排除文档及理由。
   - 受影响流程、关键章节、状态/配置/通知矩阵。
   - 可能涉及的节点/Handler/服务。
   - 需求与哪些既有功能相关。
5. 停下来确认定位是否准确。

### ⛔ 知识库不存在时的强制停止规则

当出现以下任一情况时，**必须立即停止阶段1，在 `01-定位.md` 写明原因，并告知用户，等待下一步指示**：

- MCP `get-business-rule` 返回"业务规则目录为空或不存在"（知识库未初始化）
- `shcj-common/.dev-standards/knowledge/` 目录不存在或为空

产物中必须写明：

```markdown
## 阶段1定位结果

- MCP get-business-rule：调用失败，原因：知识库未初始化（返回"业务规则目录为空或不存在"）
- 本地 .dev-standards 知识库：不存在，无业务文档/codemap/procdefs 可读
- 结论：无法完成标准定位流程，已停止。请用户告知后续处理方式（例如：直接基于需求文档和接口文档进入阶段2，或先初始化知识库）。
```

**严禁在此情况下做任何替代动作**，包括但不限于：

- ❌ 扫描并读取项目活源码（`.java`、`.xml`、`.yml` 等）来"构建代码地图"
- ❌ 读取 DAO/Service/Application 等实现类来"分析现有逻辑"
- ❌ 把活源码分析结果伪装成 codemap 写入 `01-定位.md`
- ❌ 以"帮助用户了解现状"为由继续读代码

> **原因**：同步开发中，最新代码在开发人员本机，远程仓库中的代码是不完整的早期骨架，不代表最终实现。读取并分析这些代码会得出错误结论，反而误导测试用例设计。

## 阶段2：变更分析（需求 vs 现状 vs MCP服务知识）

1. 把需求文档/联动文档描述的目标，与阶段1业务文档的现状逐项对比。
2. 如涉及服务/接口边界，调用 MCP `get-service-knowledge` 查询相关服务知识；服务名不确定时，先从 codemap、接口文档、包名、Handler 名称推断，无法确定则说明未调用原因。
3. 输出到 `.test-standards/<版本>/output/02-变更分析.md`，用表格列变更清单：
   - 新增了什么：新功能/新接口/新枚举值/新状态/新通知。
   - 修改了什么：已有规则/配置/状态迁移的改动。
   - 影响哪些老分支：哪些既有规则、Handler、通知、状态机可能被波及。
   - 三方接口变更点：接口契约、字段、证书/签名逻辑、异常码、超时、幂等。
   - MCP服务知识补充：服务职责、异常处理、幂等、已知风险。
4. 若需求、本地业务文档、MCP结果互相矛盾，必须单独列「冲突/疑点」并让用户确认。
5. 停下来让用户校正变更清单。

## 阶段3：测试概要（门禁 + 6维度测试点）

> 关键概念：测试点 ≠ 测试用例。测试点是覆盖目标；阶段4再裂变成多条原子用例。本阶段只列测试点，不写详细步骤。

进入阶段3前必须通过门禁：

```markdown
## 阶段3前置门禁

- [ ] 已读取需求文档和接口文档
- [ ] 已读取 `prodword_pic/` 图片，并重点解析“流程图/业务流程/时序图/状态流转”等字眼附近图片
- [ ] 已读取阶段2确认后的变更清单
- [ ] 已读取匹配业务流程文档全文
- [ ] 已读取对应 codemap 代码地图
- [ ] 必要时已读取 procdefs 流程定义
- [ ] 已调用 MCP get-business-rule，或说明未调用原因
- [ ] 如涉及服务/接口边界，已调用 get-service-knowledge，或说明未调用原因
- [ ] 已处理需求/业务文档/MCP结果之间的冲突或疑点
```

门禁不满足时，禁止直接生成 `03-测试概要.md`。

基于确认后的变更清单、业务文档、代码地图和流程图图片解析结果，输出到 `.test-standards/<版本>/output/03-测试概要.md`，按以下 6 个维度列测试点。每个测试点必须标注：覆盖变更项、MCP/业务规则来源、接口字段或 Handler 风险点、流程图/图片依据（如有）、预估可裂变用例数。

1. **新功能正向**：新增/修改功能的主流程。
2. **配置组合（判定表）**：对照业务文档配置矩阵做组合；维度多时用 pairwise，再补关键业务组合。
3. **状态迁移**：对照业务文档状态管理、流程定义和 Handler，覆盖正常迁移 + 异常迁移（退回 back / 拒绝 reject）。
4. **逐步骤功能操作偏离（重点）**：把目标流程拆成有序步骤，并补每个节点前置操作。流程拆分必须同时参考正文、业务文档、流程定义和需求图片中的流程图/时序图/泳道图；若图片展示了正文未写清的分支、角色或节点，要纳入矩阵或列为疑点。对每一步套用 7 类：前置/必填未满足、数量与上限、增删改+重做、操作顺序/路径偏离、重复提交、取消中断重入、取值边界。只做功能测试，不做安全越权/网络渗透/并发压测。每条都要断言最终实际落库与界面一致。
5. **接口与数据边界**：三方接口异常码、字段级空值/格式/枚举/条件必填、报数与开户失败兜底、响应字段映射。
6. **回归影响**：读阶段1代码地图和 MCP服务知识，列出改动涉及 Handler 反查出的必须回归既有功能，重点看风险区域/修改热点。

维度4必须输出「步骤 × 操作偏离类别」矩阵：行=流程步骤/前置操作，列=7类，单元格填反向测试点；不适用标 N/A 并说明原因。

停下来让用户 review，尤其确认逐步骤偏离矩阵有没有漏步骤、漏操作。

### 阶段3结束前必须输出「维度用例下限预算表」

阶段3产物末尾必须附上以下预算表，供阶段4自检使用。阶段4输出的各维度实际用例数不得低于对应下限值：

```markdown
## 维度用例下限预算表（阶段4自检基准）

| 维度 | 来源 | 计算方式 | 下限值 |
|---|---|---|---|
| 维度2-判定表 | 阶段3各判定表 | 所有判定表的有效条件组合行数之和 | ? 条 |
| 维度4-偏离矩阵 | 阶段3各步骤×偏离矩阵 | 所有矩阵中非N/A单元格总数 | ? 条 |
| 维度5-错误码 | 三方接口文档业务错误码 | 有处理逻辑的错误码总数（排除标注"不存在"的） | ? 条 |
| 维度5-落库字段 | 三方接口文档响应参数 | 标注"保存"或需落库的响应字段总数 | ? 条 |
| 维度5-通知/邮件 | 需求文档告警/邮件规则 | 通知条数 × 每条的验证要素数（标题/收件人/正文字段/附件） | ? 条 |
| 总计下限 | — | 阶段3测试点总数 × 2 | ? 条 |
```

⚠️ 阶段4开始前必须先填写此预算表的下限值，并在阶段4结束后对照自检。

## 阶段4：详细用例 + 回归清单

> 铁律：一条用例 = 单路径 + 单组数据 + 单个明确预期。严禁 1 个测试点对 1 条用例，严禁一条用例里塞多个校验点。

### 用例裂变规则

- 正反例：正向路径 1 条 + 每一条失败/异常路径各 1 条。
- 等价类 + 边界值：每个关键输入字段，有效值 / 无效值 / 边界值 各成一条。
- 枚举逐值：枚举类输入每个取值至少一条。
- 接口字段级：每字段按必填缺失 / 类型格式错 / 枚举越界 / 条件必填触发拆分；一条只破坏一个字段。
- 多步骤流程：每个独立校验点、每次接口调用、每个 UI 规则各成一条。

**维度专项裂变铁律（不得跳过）**

- **判定表逐行裂变（维度2）**：阶段3输出的每张判定表，每一有效条件组合行必须在阶段4单独成一条用例；禁止合并多行或跳过任何一行。以阶段3「维度用例下限预算表」中维度2的值为下限，未达到不得提交。

- **偏离矩阵逐单元格裂变（维度4）**：阶段3中每张「步骤 × 偏离类别」矩阵，每个非N/A单元格至少产出一条独立用例；禁止跨步骤或跨偏离类别合并单元格。以阶段3「维度用例下限预算表」中维度4的值为下限，未达到不得提交。

- **接口与数据边界全量裂变（维度5）**，以下四类必须逐项展开，不得遗漏：
  1. **错误码逐值**：三方接口文档中每个有处理逻辑的业务错误码（排除标注"不存在"的）各成一条，用例步骤触发该错误码，预期结果写明系统响应行为（自动重试/前端提示/进异常台账/人工介入等）。
  2. **响应字段落库逐字段**：接口文档中标注"保存"或需落库的每个响应字段，各成一条用例，断言该字段值正确落库（值来源正确、格式正确、不为空）。
  3. **业务通知/邮件要素逐条**：需求文档中提及的每条告警邮件/系统通知，必须按「标题、收件人、正文关键字段、附件（如有）」各成一条独立用例；通知类的用例不得以一条"验证邮件发送成功"笼统覆盖。
  4. **固定值参数验证**：接口文档中标注"固定值"的必选参数，各成一条用例，验证实际传参值与规定固定值一致。

### 输出要求

1. 按下方「内置用例模板」写入 `.test-standards/<版本>/output/04-测试用例.md`，执行阶段4时不要再读取外部模板文件。
2. 所有用例必须使用统一表头：`用例等级 | 所属模块 | 用例标题 | 前置条件 | 用例步骤 | 预期结果`。
3. `用例等级` 只允许 `1`、`2`、`3`、`4`；`所属模块` 支持按 `/` 分层级，例如 `建档/自主建档/线上授权书`，用于后续转 XMind。
4. `前置条件` 非必填；如有，转成 XMind 后作为备注放到用例标题节点下。
5. 预期结果必须唯一且可断言：状态断言引用业务文档状态枚举；通知断言引用消息通知场景编码；接口断言写明期望出参/异常码；DB/报数包断言写最终落库字段。
6. 覆盖规则、字段、MCP来源、Handler 等追溯信息优先融入 `所属模块`、`用例标题`、`用例步骤` 或 `预期结果` 中；不要额外增加表格列，避免影响 XMind 转换。
7. 单独汇总回归清单：改动点 → 影响范围 → 风险等级。
8. **自检（禁止跳过，逐项核查后方可提交）**：

   | # | 自检项 | 检查方法 | 通过标准 |
   |---|---|---|---|
   | ① | 维度2判定表行数覆盖 | 统计阶段3各判定表有效行数之和，与本次维度2用例数对比 | 实际用例数 ≥ 预算下限 |
   | ② | 维度4矩阵单元格覆盖 | 统计阶段3所有矩阵非N/A单元格总数，与本次维度4用例数对比 | 实际用例数 ≥ 预算下限 |
   | ③ | 维度5-错误码覆盖 | 列出接口文档所有有效业务错误码，逐一确认已各成独立用例 | 所有有效错误码均有对应用例 |
   | ④ | 维度5-落库字段覆盖 | 列出接口文档标注"保存"的响应字段，逐一确认已各成落库验证用例 | 所有需落库字段均有对应用例 |
   | ⑤ | 维度5-通知/邮件要素覆盖 | 列出需求文档中所有通知/邮件场景，确认已按要素（标题/收件人/正文字段/附件）各成用例 | 无通知场景被笼统覆盖 |
   | ⑥ | 总量下限 | 用例总数与阶段3「维度用例下限预算表」中总计下限对比 | 总用例数 ≥ 阶段3预算总计下限 |

   以上任一项结果为"未通过"，必须返工补齐，禁止直接输出。

### 内置用例模板

阶段4生成 `04-测试用例.md` 时，直接使用以下模板：

```markdown
# 测试用例 - <版本>

> 需求来源：`.test-standards/<版本>/input/prodword/`  
> 参考资料：`.test-standards/<版本>/input/reference/`  
> 输出位置：`.test-standards/<版本>/output/04-测试用例.md`

## 表格规范

必填字段：

- `用例等级`：只允许 `1`、`2`、`3`、`4`。
- `所属模块`：支持按 `/` 分层级，后续转 XMind 使用。例如 `建档/自主建档/线上授权书`。
- `用例标题`：单条原子用例标题，标题应能独立表达校验目标。
- `用例步骤`：单路径、单组数据、单个明确动作链；多步用 `<br>` 或 `1. 2. 3.` 分隔。
- `预期结果`：唯一明确预期，必须可断言。

非必填字段：

- `前置条件`：如有，转成 XMind 后作为备注放到用例标题节点下。

## 用例表

| 用例等级 | 所属模块 | 用例标题 | 前置条件 | 用例步骤 | 预期结果 |
|---|---|---|---|---|---|
| 1 | 一级模块/二级模块 | 示例：线上授权书未完成倒计时不允许下一步 | 已进入线上授权书预览页 | 1. 打开授权书预览页<br>2. 倒计时未结束时点击【下一步】 | 【下一步】不可点击或提示需完成阅读；不生成阅读日志；不进入后续报数 |

## 填写要求

- 一条用例只覆盖一个明确校验点，禁止一条用例塞多个异常字段或多个分支。
- `所属模块` 必须能形成 XMind 层级，不要为空。
- `用例标题` 中不要手写等级，转 XMind 脚本会自动把等级标到标题上。
- `用例步骤` 和 `预期结果` 内如需换行，优先使用 `<br>`。
```

### 转 XMind

阶段4完成后，可用脚本把 `.test-standards/<版本>/output/04-测试用例.md` 转成 XMind：

```bash
python ".cursor/skills/testcase-gen-mcp/scripts/cases_to_xmind.py" ".test-standards/<版本>"
```

也可以显式指定输入和输出：

```bash
python ".cursor/skills/testcase-gen-mcp/scripts/cases_to_xmind.py" ".test-standards/<版本>/output/04-测试用例.md" ".test-standards/<版本>/output/<版本>.xmind"
```

XMind 层级规则：

- 根节点：版本目录名称，例如 `V1.12.0-xxx`。
- 中间节点：`所属模块` 按 `/` 拆分形成层级。
- 用例节点：`用例标题`，脚本自动加等级前缀，如 `【等级1】xxx`。
- 用例节点下包含：`用例步骤`、`预期结果`。
- `前置条件` 如有，写入用例节点备注。

## 注意事项

- 全程中文。术语与业务文档保持一致，如 `CUST_BUILD` / `CUST_CHANGE` / `BUILD_ACTIVATE`。
- 知识库与代码只读，不修改 `shcj-common/.dev-standards` 和业务源码。
- MCP结果是辅助证据，不是最终真相；必须和本地文档、代码地图、用户确认结论交叉校验。
- 若业务文档、需求、MCP结果出现矛盾，明确指出并提示用户确认。
- 阶段产物要写清楚来源，避免“AI拍脑袋”的测试点。
- 需求文档图片可能包含版本迭代流程图。阶段0必须抽取并读取，阶段3必须把流程图中的角色、节点、分支、状态流转和接口调用顺序纳入测试概要设计。
- **不得以活源码替代任何知识库文档**。`codemap`、`business`、`procdefs` 均指 `.dev-standards` 下的**静态文档**。当这些文档不存在时，必须停下来告知用户，等待指示，绝不自行读取 `.java`/`.xml`/`.yml` 等活源码作为替代。
