## 一、Word 转 Markdown

目标：把 `input/<需求名>/**/*.docx|*.doc` 转成同名 `.md`，保留原文件。优先用已安装的工具，按下面顺序尝试，成功即可。

### .docx
1. **pandoc（首选）**：
   ```bash
   pandoc "输入.docx" -o "输入.md" --extract-media=./media
   ```
2. pandoc 不可用时，用 Python `mammoth`（更好地保留语义）或 `python-docx`：
   ```bash
   pip install mammoth
   python -c "import mammoth,sys;print(mammoth.convert_to_markdown(open(sys.argv[1],'rb')).value)" "输入.docx" > "输入.md"
   ```

### .doc（旧二进制格式，pandoc 不直接支持）
1. 用 LibreOffice 先转 docx 再转 md：
   ```bash
   soffice --headless --convert-to docx "输入.doc" --outdir "."
   ```
   然后按 .docx 流程处理。
2. 上述都不可用时：提示用户「.doc 缺少转换工具，请另存为 .docx 或 .md 后重试」，不要硬猜内容。

### 转换后自检
- 表格是否完整（需求里的配置/字段表格最重要，丢了要重转）。
- 是否有乱码（编码问题）。
- 图片转成了引用即可，看不到图时在对应位置标注「[图片，需人工查看原文件]」。

以下两点是 SKILL.md 阶段3维度3未涵盖的补充，其余基本原则（正常迁移+异常迁移）已在 SKILL.md 中说明。

- 重点覆盖「退回后重新提交」「拒绝后释放/置失败」这类反向迁移，这是最容易漏的场景。

## 三、接口测试补充清单（CFCA / CA 等对接重点）

SKILL.md 阶段4维度5已覆盖：错误码逐值、请求参数传参逻辑、响应字段落库、业务通知要素。
以下4项是维度5的补充，需在维度5用例完成后逐项自查：

- **超时**：接口超时时本侧流程的表现（是否挂起、是否可重试、状态是否正确）。
- **重试 / 幂等**：重复调用是否产生重复数据（如重复注册 CA、重复建档）；重复提交时系统是否有幂等保护。
- **签名 / 证书**：CFCA 场景关注证书有效期、签名验签失败、证书未注册等分支。
- **联动一致性**：三方返回后，本侧建档状态 / 客户状态 / 产品开通状态是否**同步正确**（跨系统状态一致性，不仅是本侧落库）。

## 四、用例裂变（测试点 → 多条原子用例）

> ⚠️ 本节保留供对照参考，待与 SKILL.md 阶段4用例裂变规则完成整合后可删除。
> SKILL.md 的版本已更完整（含高风险判断标准、维度专项铁律、合并说明清单机制），执行时以 SKILL.md 为准。

核心：**测试点是覆盖目标，用例是单路径+单数据+单预期的原子条目。一个测试点几乎不可能只对应一条用例。**

裂变维度（对每个测试点逐一套用）：

| 维度 | 怎么裂 | 例子 |
|---|---|---|
| 正反例 | 正向 1 条 + 每条失败路径各 1 条 | 法人核验：通过 1 条 + 不通过 1 条 |
| 等价类 | 有效类 / 各无效类各一条 | custType=COMPANY(有效) / =PERSON / 为空 |
| 边界值 | 上界、下界、越界各一条 | 字段长度、金额、日期 |
| 枚举逐值 | 每个枚举值一条 | certStatus：SUCCESS/NEW_APPLY/FAIL/INVALID 各一条 |
| 条件必填 | 触发条件 / 不触发 各一条 | 法人证件=身份证→报四要素；≠身份证→报二要素 |
| 接口字段 | 每字段 ×（缺失/格式/枚举/条件）各一条，一条只破坏一个字段 | appId 缺失一条、sign 缺失一条、authTime 格式错一条… |
| 流程步骤 | 每个独立校验点一条 | 一证四步：工商/法人/经办人/倒计时/报数/注册 各一条 |

**反例（错误，1条塞多个校验）**：
> TC-306：appId空、sign空、custType错、subjectType错 → 预期失败

**正解（裂成4条，各自单一预期）**：
> TC-306a：appId 缺失 → 接口拒绝并提示 appId 必填
> TC-306b：sign 缺失 → 接口拒绝并提示验签失败
> TC-306c：custType=PERSON（企业场景）→ 拒绝/类型不符
> TC-306d：subjectType 与核验对象不符 → 拒绝

## 五、功能操作偏离场景清单（逐步骤、逐前置操作套用）

> 范围：**只做功能测试，不做安全/越权/网络渗透/并发压测类**（除非用户另行要求）。
> 重点：围绕每个步骤的**操作偏离**和**操作序列**，并且**每条用例都要断言最终实际落库/状态与界面是否一致**。

用法：把目标流程拆成**有序步骤**，为每个节点补上**前置操作**（进入该节点前该做的事），然后对**每一个步骤**逐条过下面 7 类，列出适用的功能反向测试点。阶段3填矩阵时用这7类的"该问什么"来识别场景；阶段4写用例时用"断言重点"来写预期结果。

| # | 类别 | 该步骤要问的问题（举例，均为功能层面） |
|---|---|---|
| 1 | 前置/必填未满足就推进 | 该上传不上传就点下一步？该勾选不勾选就提交？必填留空？前置步骤没做完就进入本步？前一节点资料缺失时本节点表现如何？ |
| 2 | 数量与上限 | 只能传 1 个却传多个？超出数量上限？重复添加同一项？最少数量（传 0 个）？明细/列表条数上下限？ |
| 3 | 增删改 + 重做（**核心**） | 上传后删除？删除后重新上传？替换/覆盖已传文件？反复增删几轮？改了又改？→ **断言最终落库内容、数量、版本与最后一次操作一致，不残留旧数据** |
| 4 | 操作顺序 / 路径偏离 | 步骤乱序？返回上一步再前进，已填数据是否还在？跳步？中途切换互斥选项（如线上↔线下、四要素↔二要素）后，旧数据是否被正确清空/覆盖？ |
| 5 | 重复 / 多次提交 | 重复提交同一步？提交成功后再回来改再提交？连续多次发起？→ 断言不产生重复数据 |
| 6 | 取消 / 中断 / 重入 | 中途返回/关闭弹窗/放弃？未完成就退出，再进来是续填草稿还是清空？关弹窗后重新进入数据是否还在？ |
| 7 | 取值边界（功能）**⚠️ 不进阶段3矩阵** | 此类属于数据取值测试，由阶段4裂变规则（等价类/边界值/枚举逐值/条件必填）展开，不在矩阵列中体现。识别时可问：该步骤有哪些输入字段存在格式/边界/枚举约束？→ 记录后交阶段4处理。 |

**节点前置操作专项**（最容易漏，逐节点单独想）：进入某节点前，正常应完成的操作如果**没做 / 做一半 / 做错再改**，本节点和后续落库会怎样。例：
- 「确认CA开通信息」节点前，企业资料未传齐就进入；
- 上传资料节点：不传就下一步 / 传多份 / 传后删除再传 / 传错类型删掉重传，最终落库以哪份为准；
- 切换签署方式（线上↔线下）后，之前生成的报数包/授权书数据是否被正确清理。

**断言重点（阶段4每条用例必带）**：操作后查 DB 实际落库——删除是否真删、重传是否覆盖旧值、最终保存的是不是最后一次有效操作的结果、字段值与界面所选一致。

**自检**：每个有"输入/上传/选择/多步"的步骤，至少应在第 1~5 类里产出反向测试点；只有正向一条说明没展开，需返工。

## 六、回归影响分析（用代码地图）

对应 SKILL.md 阶段4输出要求第7条，具体方法如下：

1. 从变更清单确定改动落在哪些节点 / Handler。
2. 打开对应 `codemap` 代码地图，定位这些 Handler，看它们还被哪些流程/节点复用。
3. 重点查代码地图的「风险区域」「修改热点」段落（如额度占用释放、运营中台幂等、事务完整性）。
4. 输出：改动点 → 受影响功能 → 风险等级 → 必回归用例。
